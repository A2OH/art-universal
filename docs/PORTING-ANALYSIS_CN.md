# ART移植分析：三种策略实现10-50倍性能提升

## 1. ART架构

```
┌─────────────────────────────────────────────────────────┐
│                    ART运行时（623K行C++）                  │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  解释器（2,010行）                                  │   │
│  │  - 基于switch的字节码分发                           │   │
│  │  - 632行内建函数（Math、String、Array）             │   │
│  │  - 比Dalvik解释器快3-5倍                           │   │
│  └───────────────────────────────────────────────────┘   │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  优化编译器（50K行）                                │   │
│  │  DEX→IR → 15轮优化 → 代码生成器（ARM32/64 x86/64）│   │
│  └───────────────────────────────────────────────────┘   │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  运行时服务                                        │   │
│  │  类链接器(9,883行) | GC(9种收集器) | JNI(12,561行) │   │
│  └───────────────────────────────────────────────────┘   │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  dex2oat（10K行）— 离线AOT编译工具                  │   │
│  └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 关键依赖（全部为POSIX标准）

| 库 | 用途 | OHOS可用？ |
|----|------|:---:|
| libz | 压缩 | 是 |
| liblog | 日志 | 桩化或重定向 |
| libicuuc | Unicode | 是（OHOS有ICU） |
| pthreads | 线程 | 是（POSIX） |
| mmap/mprotect | 内存管理 | 是（POSIX） |

**无LLVM。无Android Binder。无专有库。**

## 2. 策略A：仅AOT（dex2oat）

### 工作原理

```
构建时（在主机x86_64上）：
  classes.dex → dex2oat → classes.oat（原生ARM代码）

运行时（在OHOS ARM32上）：
  加载classes.oat → 直接执行原生代码
  无需解释器（除极少数回退）
```

### dex2oat的15轮优化

1. 方法内联（内联小方法）
2. 常量折叠（编译时计算常量）
3. 死代码消除（删除未使用分支）
4. 寄存器分配（映射到CPU寄存器）
5. 空检查消除（证明引用非空）
6. 边界检查消除（证明索引在范围内）
7. 循环优化（展开、强度削减）

### 移植工作量

| 组件 | 代码行 | 周数 | 备注 |
|------|------:|:----:|------|
| dex2oat工具 | ~10K | 2 | 在主机Linux上运行 |
| 编译器IR+优化 | ~30K | 0 | 已与架构无关 |
| ARM32代码生成器 | ~20K | 0 | 已存在 |
| .oat文件加载器 | ~5K | 2 | ELF读取+重定位 |
| 基本GC | ~10K | 2 | 标记-清除 |
| 类链接器 | ~10K | 2 | 类加载+解析 |
| **合计** | **~93K** | **10-12** | **2-3个月** |

## 3. 策略B：仅ART解释器

### 为什么比Dalvik快3-5倍

| 特性 | Dalvik | ART解释器 | 加速 |
|------|--------|----------|-----:|
| 分发方式 | switch(opcode) | 计算goto | 1.5-2倍 |
| 内建函数 | 无 | Math.abs等→内联 | 5-50倍/次调用 |
| 字段访问 | 哈希表查找 | 直接偏移（链接时解析） | 3-5倍 |
| 虚方法分发 | 每次vtable查找 | 内联缓存（记住上次类型） | 2-3倍 |

### 移植工作量

| 组件 | 代码行 | 周数 | 备注 |
|------|------:|:----:|------|
| ART解释器 | ~15K | 2 | interpreter_switch_impl-inl.h |
| 内建函数 | ~2K | 1 | interpreter_intrinsics.cc |
| 运行时核心 | ~20K | 3 | 线程、类链接器（简化）、JNI |
| 基本GC | ~10K | 2 | 标记-清除 |
| **合计** | **~49K** | **9** | **约2个月** |

### 为什么从这里开始

- **最低风险** — 无编译器、无代码生成、无反优化
- **直接替换** — 相同DEX输入、相同JNI接口
- **立即收益** — 仅内建函数就在Math/String/Array操作上提供5-50倍加速

## 4. 策略C：完整ART（解释器 + JIT + AOT）

### 为什么JIT难移植

| 挑战 | 原因 | 工作量 |
|------|------|:------:|
| `mmap(PROT_EXEC)` | 运行时将原生代码写入可执行内存。OHOS安全策略可能限制 | 2周 |
| 信号处理（SIGSEGV） | ART将地址0映射为不可访问，通过信号捕获空引用解引用 | 2周 |
| 反优化 | JIT编译方法的假设被打破时，必须在执行中途展开原生栈并恢复到解释器——最难的部分 | 3周 |
| 代码缓存GC | JIT生成的代码需要与托管堆GC协调的特殊内存区域管理 | 1周 |
| 线程暂停 | JIT需要停止所有线程进行代码修补和GC | 2周 |

### 移植工作量

| 组件 | 代码行 | 周数 |
|------|------:|:----:|
| 策略A+B全部内容 | ~93K | 10-12 |
| JIT编译器集成 | ~15K | 4 |
| 信号处理 | ~3K | 2 |
| 反优化 | ~5K | 3 |
| **合计** | **~124K** | **22-24（5-6个月）** |

## 5. 推荐分阶段路径

```
第1-2月：策略B — ART解释器
         └── 3-5倍加速，最低风险，验证运行时移植

第2-4月：策略A — 添加dex2oat AOT
         └── 预编译代码10-50倍加速
         └── 在主机上运行dex2oat，部署.oat到设备

第4-6月：策略C — 添加JIT（可选）
         └── 未被AOT覆盖的代码的动态优化
```

## 6. RISC-V考虑

ART Android 11无RISC-V代码生成器。选项：
1. 先写ARM32代码生成器，暂忽略RISC-V
2. 使用Android 14+ ART（有初步RISC-V支持）
3. 编写RISC-V代码生成器（约20K行，3个月）

## 7. 构建系统

ART使用Android的soong（Android.bp）。独立构建选项：
- 转换为CMakeLists.txt（约2周）
- 转换为GN（Chrome/OHOS使用，约1周）
- 编写Makefile（最简单，约3天）
