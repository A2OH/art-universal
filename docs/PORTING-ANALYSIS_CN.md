# ART移植分析：三种策略实现10-50倍性能提升

## 1. ART架构

```
┌─────────────────────────────────────────────────────────┐
│                    ART Runtime (623K行C++)                │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  解释器 (2,010行)                                  │   │
│  │  - 基于switch的字节码分发                           │   │
│  │  - 632行内建函数 (Math, String, Array)              │   │
│  │  - 比Dalvik解释器快3-5倍                            │   │
│  └───────────────────────────────────────────────────┘   │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  优化编译器 (50K行)                                │   │
│  │  ┌─────────┐  ┌──────────┐  ┌──────────────────┐  │   │
│  │  │ DEX→IR  │→ │  优化    │→ │ 代码生成器       │  │   │
│  │  │ (节点)  │  │ (15 pass)│  │ ARM32/64 x86/64  │  │   │
│  │  └─────────┘  └──────────┘  └──────────────────┘  │   │
│  └───────────────────────────────────────────────────┘   │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  运行时服务                                        │   │
│  │  类链接器 (9,883行) — 类加载                       │   │
│  │  GC (9种收集器) — 并发标记清除                      │   │
│  │  JNI (12,561行) — 本地方法桥接                     │   │
│  │  线程管理 — 基于pthread                            │   │
│  └───────────────────────────────────────────────────┘   │
│                                                           │
│  ┌───────────────────────────────────────────────────┐   │
│  │  dex2oat (10K行) — 离线AOT编译工具                 │   │
│  │  输入: classes.dex → 输出: 本地代码.oat文件         │   │
│  └───────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 关键依赖（全部POSIX标准）

| 库 | 用途 | OHOS上可用？ |
|---|------|:---:|
| libz | 压缩 (zlib) | 是 |
| liblog | 日志 | 桩化或重定向 |
| libbacktrace | 栈回溯 | 移植或桩化 |
| libicuuc | Unicode (ICU) | 是（OHOS有ICU） |
| libdl | 动态加载 | 是（musl有dlopen） |
| libartpalette | 平台抽象 | 桩化（约20个方法） |
| pthreads | 线程 | 是（POSIX） |
| mmap/mprotect | 内存管理 | 是（POSIX） |

**无LLVM。无Android Binder。无专有库。**

## 2. 策略A：仅AOT（dex2oat）

### 工作原理

```
构建时（在宿主机x86_64上）：
  classes.dex → dex2oat → classes.oat（本地ARM代码）

运行时（在OHOS ARM32上）：
  加载classes.oat → 直接执行本地代码
  不需要解释器（极少情况回退除外）
```

### dex2oat做什么

1. 解析DEX文件 → 提取方法
2. 构建HGraph IR（中间表示）
3. 运行15个优化pass：
   - 内联（内联小方法）
   - 常量折叠（编译时求值常量）
   - 死代码消除（移除未使用分支）
   - 寄存器分配（映射到CPU寄存器）
   - 空指针检查消除（证明引用非空）
   - 边界检查消除（证明索引在范围内）
   - 循环优化（展开、强度削减）
4. 通过架构特定代码生成器生成本地代码
5. 输出.oat文件（类ELF格式，包含本地代码和元数据）

### 移植工作量

| 组件 | 行数 | 周数 | 备注 |
|------|------:|:-----:|------|
| dex2oat工具 | ~10K | 2 | 在宿主机Linux运行——改动极少 |
| 编译器IR + 优化 | ~30K | 0 | 已经架构无关 |
| ARM32代码生成器 | ~20K | 0 | 已存在（code_generator_arm_vixl.cc） |
| .oat文件加载器 | ~5K | 2 | ELF读取器 + 重定位修补 |
| 运行时桩（入口/出口） | ~3K | 1 | 托管↔本地转换跳板 |
| 基础GC（标记清除） | ~10K | 2 | 使用mmap的堆管理 |
| 类链接器 | ~10K | 2 | 子集——类加载 + 解析 |
| JNI桥接 | ~5K | 1 | 从Dalvik的工作JNI适配 |
| **合计** | **~93K** | **10-12** | **2-3个月** |

### 风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| .oat格式版本耦合 | 运行时必须匹配dex2oat版本 | 两者锁定到同一AOSP提交 |
| ARM32代码生成器在OHOS上的bug | 生成错误本地代码 | 使用已有AOSP测试套件 |
| GC堆的mmap | OHOS可能限制大型映射 | 使用较小堆，多个区域 |
| 静态链接vs共享 | dex2oat期望共享库 | 为运行时构建辅助.so |

## 3. 策略B：仅ART解释器

### 为什么比Dalvik快3-5倍

| 特性 | Dalvik | ART解释器 | 加速 |
|------|--------|----------|------:|
| 分发 | `switch(opcode)` | 计算型goto（gcc扩展） | 1.5-2倍 |
| 内建函数 | 无 | Math.abs, String.length, Array.length → 内联 | 每次调用5-50倍 |
| 字段访问 | 哈希表查找 | 直接偏移（类链接时已解析） | 3-5倍 |
| 虚分发 | 每次vtable查找 | 内联缓存（记住上次类型） | 2-3倍 |
| 字符串操作 | 完整方法调用 | 内建快速路径 | 5-10倍 |

### 移植工作量

| 组件 | 行数 | 周数 | 备注 |
|------|------:|:-----:|------|
| ART解释器 | ~15K | 2 | interpreter_switch_impl-inl.h |
| 内建函数 | ~2K | 1 | interpreter_intrinsics.cc |
| 运行时核心 | ~20K | 3 | 线程、类链接器（简化版）、JNI |
| 基础GC | ~10K | 2 | 标记清除（复用策略A） |
| libc兼容性 | ~2K | 1 | OHOS musl与Android bionic差异 |
| **合计** | **~49K** | **9** | **约2个月** |

### 为什么先从这里开始

- **风险最低** — 无编译器、无代码生成、无反优化
- **直接替换** — 相同DEX输入、相同JNI接口
- **立即受益** — 仅内建函数就能对Math/String/Array操作带来5-50倍提升
- **为A/C打基础** — 运行时核心（GC、类链接器、线程）是共享的

## 4. 策略C：完整ART（解释器 + JIT + AOT）

### JIT比AOT多什么

| 能力 | 仅AOT | 带JIT |
|------|:---:|:---:|
| 首次运行前编译 | 是 | 是 |
| 运行时编译新代码 | 否 | 是——检测并编译热点方法 |
| Profile引导优化 | 否（静态猜测） | 是——真实分支频率 |
| 推测性内联 | 保守 | 激进（错误时反优化） |
| 适应用户模式 | 否 | 是——用更好的类型信息重新编译 |

### 为什么JIT难移植

| 挑战 | 为什么难 | 工作量 |
|------|---------|:------:|
| `mmap(PROT_EXEC)` | 需要在运行时将本地代码写入可执行内存。OHOS安全策略可能限制。 | 2周 |
| 信号处理器（SIGSEGV） | ART将地址0映射为不可访问，捕获空解引用作为信号而非显式检查。OHOS信号模型可能不同。 | 2周 |
| 反优化 | 当JIT编译方法的假设失效（加载新类），ART必须在执行中途展开本地栈并在解释器中恢复。这是JIT最难的部分。 | 3周 |
| 代码缓存GC | JIT生成的代码位于特殊内存区域。满时必须淘汰旧代码。需要与托管堆GC仔细协调。 | 1周 |
| 性能分析 | 方法调用计数、分支频率、类型profile——全部在运行时收集以指导JIT决策。Hook到解释器和编译代码中。 | 1周 |
| 线程挂起 | JIT需要停止所有线程进行代码修补和GC。ART使用与运行时紧密集成的检查点挂起。 | 2周 |

### 移植工作量

| 组件 | 行数 | 周数 | 备注 |
|------|------:|:-----:|------|
| A + B的所有内容 | ~93K | 10-12 | 见上 |
| JIT编译器集成 | ~15K | 4 | jit.cc, jit_code_cache.cc |
| 信号处理器 | ~3K | 2 | SIGSEGV、SIGBUS处理 |
| 反优化 | ~5K | 3 | 栈展开、帧重建 |
| 性能分析 | ~5K | 1 | 方法计数器、类型profile |
| 线程检查点 | ~3K | 2 | 挂起、安全点 |
| **合计** | **~124K** | **22-24** | **5-6个月** |

## 5. 推荐分阶段方案

```
第1-2月：策略B — ART解释器
         └── 3-5倍加速，风险最低，验证运行时移植
                
第2-4月：策略A — 添加dex2oat AOT
         └── 预编译代码10-50倍加速
         └── 在宿主机使用dex2oat，部署.oat到设备
                
第4-6月：策略C — 添加JIT（可选）
         └── 动态优化未被AOT覆盖的代码
         └── 复杂度最高，相比AOT收益递减
```

### 各策略适用的应用类型

| 应用类型 | 策略B（3-5倍） | 策略A（10-50倍） | 策略C（10-50倍+） |
|---------|:---:|:---:|:---:|
| 计算器、设置 | 足够 | 过度 | 不需要 |
| 购物、社交Feed | 可接受 | 好 | 理想 |
| 游戏、地图、相机 | 太慢 | UI够用 | 需要 |
| 实时视频/音频 | 不可行 | 可接受 | 必需 |

## 6. RISC-V相关

ART Android 11没有RISC-V代码生成器。选项：

1. **先写ARM32代码生成器，暂时忽略RISC-V** — 最简单，覆盖95%设备
2. **使用Android 14+ ART**（有初步RISC-V支持）— 但14有更多Android系统依赖
3. **编写RISC-V代码生成器**（约20K行，有经验的编译器开发者需3个月）
4. **AOT交叉编译** — 在x86_64宿主机运行dex2oat，生成RISC-V代码——只需dex2oat中的RISC-V后端，运行时不需要
